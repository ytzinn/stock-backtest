# DART 정정공시 룩어헤드 바이어스 — 현황 및 해결 방안 검토

## 문제 요약

`dart_ingest`가 과거 전체 재무데이터를 **단 한 번** 수집하는 구조이기 때문에,
수집 시점(2026년 5월) 이전에 정정된 모든 과거 데이터가 "미래 정보"를 담은 채 저장된다.

```
예시: FY2022 데이터

원본 공시일:  2023-03-15  ← available_from 에 올바르게 저장됨
FY2022 정정:  2024-06-01  ← DART API가 이미 반영한 상태
dart_ingest:  2026-05-xx  ← 이 시점에 FY2022 수집

DB에 저장된 값:
  available_from = 2023-03-15  (정확)
  amount         = 2024-06-01 정정값  (룩어헤드!)

4월 2023 리밸런싱:
  available_from <= 2023-04-03 → 조건 통과 ✓
  amount = 2024년에나 공개된 정정값 → 룩어헤드 ✗
```

---

## DART API 동작 방식 (확인된 사실)

`fnlttSinglAcntAll.json`은 **현재 시점 최신 버전**만 반환한다.

- `thstrm_amount`: 당기 — 정정이 있으면 정정 후 값
- `frmtrm_amount`: 전기 비교 — 당기 보고서 기준으로 재산정된 전기값 (소급 수정 포함 가능)

과거 원본 공시 값을 API로 되찾을 방법 없음. 공시 직후 수집해 보관하는 것만이 원본 보전의 유일한 수단.

---

## 현재 original_amount / amendment_from 설계의 실제 보호 범위

| 시나리오 | 보호 여부 | 이유 |
|----------|-----------|------|
| dart_ingest **이후** 정정 → amendment_checker 실행 | ✓ 보호 | original_amount에 수집 당시 값 보전, amendment_from 기록 |
| dart_ingest **이전**에 이미 정정된 과거 데이터 | ✗ 미보호 | 이미 정정값이 amount에 들어있고, 원본값 복구 불가 |
| 정정 없는 IFRS 전기 재산정 (연결범위·회계정책 변경) | ✗ 미보호 | frmtrm 차이로 감지되나, 원본도 정정값도 아닌 "소급 재작성" |

결론: **2026년 5월 이후 발생하는 정정공시에 대해서만 보호**. 역사적 백테스트 전 기간(2015~2025)은 정도의 차이가 있을 뿐 구조적으로 룩어헤드를 내포.

---

## 실질적 영향 규모 (FRMTRM 분석 결과)

dart_ingest가 각 연도 FY 데이터를 수집할 때, DART API가 반환하는 전기 비교값(frmtrm)과
우리 DB에 저장된 전기 값이 다른 경우를 집계한 결과:

| 구분 | 건수 |
|------|------|
| 전체 FRMTRM 경고 | 17,396건 |
| 단위 오류(×1000) 제외 후 IS 계정 실질 차이 | 1,460 그룹 |
| 2023년 이후 IS 계정 차이 | 433 그룹 |
| 단위 버그(×1,000,000) — 명백한 데이터 오류 | 4 그룹 (032680, 039230) |

### 차이 원인 추정 분류

1. **IFRS 전기 재산정** (~70%): 연결 범위 변경, 회계정책 변경으로 전기값 소급 수정.
   - 특징: 10~30% 차이, 여러 계정 동시 변동
   - 룩어헤드 여부: 논쟁적 (당시 투자자도 사업 구조 변경을 인지하고 있었을 가능성)

2. **실제 기재 오류 정정** (~20%): 입력 실수, 단위 오류 등.
   - 특징: 50%+ 차이 또는 부호 반전, 단일 계정 집중
   - 룩어헤드 여부: 명백한 룩어헤드 (수정 전 틀린 값이 당시 공개 정보)

3. **명백한 데이터 버그** (~10%): 032680처럼 ×1,000,000 오류.
   - 처리: DB 직접 수정 완료

## 해결 방안 후보

### 방안 A: 한계로 인정하고 현행 유지

**근거:**
- Compustat, Bloomberg Terminal 등 상용 데이터도 동일 문제 내포 (point-in-time 재구성 제품은 별도 판매)
- 학술 팩터 백테스트 문헌의 대부분은 이 한계를 명시하고 진행
- IFRS 재산정 차이는 "오류"가 아니므로, 원본값 사용이 반드시 더 정확한 것도 아님
- 가장 영향 큰 계정(매출액)의 대형주 오류는 드물고, 소형주는 팩터 점수 영향 제한적

**단점:**
- 역사적 백테스트 성과가 실제보다 좋게 나올 수 있음 (정정으로 개선된 종목을 미리 선택)
- SPEC에 명시적 한계로 기록 필요

---

### 방안 B: dart_ingest 증분 방식 재설계 (근본 해결)

매일 또는 공시 즉시 수집 → 당시 원본값 보전 → 정정 발생 시 정정값을 새 버전으로 추가.

```
disclosures 테이블: 공시 이벤트 (원본/정정 구분)
financials_versions 테이블: (ticker, year, report_type, rcept_no, account_nm, amount)
  → 각 공시 시점의 스냅샷

백테스트 쿼리: rebalance_date 기준으로 당시 최신 버전 선택
```

**장점:** 완전한 PIT 보장
**단점:**
- 현재 과거 데이터는 원본 복구 불가 → 2026년 5월 이전 기간은 여전히 오염
- 개발 공수 큼 (스키마 재설계, 백테스트 쿼리 전면 수정)
- DART API 일일 한도 10,000콜 압박 (dart-watcher 공유)

---

### 방안 C: frmtrm 역추산으로 부분 보정

FY(N) 수집 시 frmtrm은 "FY(N-1)의 현재 시점 비교값". 이를 이용해 FY(N-1) 저장값 보정.

```
예: FY2023 수집 시
  frmtrm['매출액'] = 500억  (DART가 제공하는 2022년 비교값)
  우리 DB FY2022['매출액'] = 800억  → FRMTRM_MISMATCH 38% 차이

frmtrm으로 역보정:
  financials_frmtrm[2022]['매출액'] = 500억
  → FY2023 보고서 기준의 2022년 비교값이므로, FY2023 수집 시점 이후 기준으로는 정확
```

**장점:** 추가 API 호출 없이 기존 수집 데이터 활용 가능
**단점:**
- frmtrm 자체가 "FY(N) 관점에서 재산정한 FY(N-1)" → 또 다른 버전 혼용
- FY(N) 수집 이전 리밸런싱에는 적용 불가 (frmtrm도 미래 정보)
- 구현 복잡도 상당

---

### 방안 D: 수집 시점 기준 available_from 재정의 (현실적 타협)

"원본 공시일"이 아니라 **dart_ingest 실행일** 기준으로 available_from을 재설정.

```
현재: available_from = MIN(rcept_dt) = 2023-03-15
수정: available_from = dart_ingest 실행일 = 2026-05-20

효과:
  → 2026-05-20 이전 모든 리밸런싱에서 해당 종목 데이터 미사용
  → 사실상 2015~2025 역사적 백테스트 불가
```

**단점:** 백테스트 기간이 사라지므로 사실상 의미 없음

## 현재 시스템 실제 사용 권고

### 단기 (현행 유지 + 한계 명시)

- 역사적 백테스트(2015~2025)는 **정정공시 룩어헤드 내포** 사실을 SPEC에 기록
- 영향이 큰 케이스(FRMTRM 50%+ 차이)만 선별적으로 수동 검토
- amendment_checker는 **2026년 이후 신규 정정 보호** 목적으로만 운영

### 중기 (증분 수집 전환)

- dart_ingest를 cron 기반 **증분 수집** 으로 전환
- 2026년 5월 이후부터는 완전한 PIT 보장
- 역사적 데이터는 한계 인정, 2026년 이후 실전 운영에서 정확도 확보
