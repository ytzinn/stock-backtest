## 1. 무엇이 잘못됐었나

`overlay_returns`(tilt 모드)의 `base_return`(always-on 비교군) 컬럼이 `base_series.asof(d)` — always-on **월별** 시리즈에서 해당 시점 이전 가장 가까운 **1개월치 값**만 가져오는 방식으로 계산되고 있었다.

문제는 같은 행의 `net_port_return`은 `nav_path()`로 **그 행이 실제로 커버하는 구간**(월간 1개월 / 분기 3개월 / 반기 4~8개월)을 정확히 복리 계산한 값이라는 것 — 즉 `overlay_freq=quarterly`나 `semiannual`에서는 **3~8개월 복리 수익 vs 1개월 스냅샷**을 그대로 빼서 "알파"라고 부르고 있었다.

추가로 `tilt_option=B_two_sided`(s_neutral=0.8)에서는 `base_return` 자체가 이미 대체 sleeve와 20% 블렌딩된 값이라, "순수 소형가치(s=1.0)"라는 전제로 설계된 `SPEC_08_B05` 분해 실험의 재조합 공식과도 어긋났다.

발견 경위: `SPEC_08_B05_timing_vs_deconcentration.md`(타이밍/탈집중 분해 실험) 구현 전 검토 과정에서, 그 문서가 `base_return`을 "소형 value sleeve 수익(s=1.0)"으로 가정하고 있는데 실제 스키마/코드와 대조하다가 발견.

## 2. 수정 내용

- **B-FIX-1**: `overlay_returns`에 `small_return`(순수 소형가치, `nav_path()` 구간 복리) 컬럼 신설. `_period_tilt_rows()`가 이미 계산해두고 버리던 `small_ret` 지역변수를 저장하도록 변경.
- **B-FIX-2**: `base_return`을 `base_series.asof(d)` 대신 `s_neutral·small_return + (1−s_neutral)·alt_return`로 재계산. `small_return`/`alt_return` 둘 다 이 행과 정확히 같은 구간을 `nav_path()`로 이미 복리 계산해둔 값이라 **추가 DB 조회 없이** 순수 산술만으로 해결됨.
- 커밋: `ec17c99` (`phase-b-signal-tilt` 브랜치, 서버 반영 완료)
- 로컬 테스트 71개 전체 통과(신규 회귀 테스트 2개 포함 — s_neutral=1.0 항등성, B_two_sided+semiannual 구간 블렌딩 검증)

## 3. 재실행 및 신구 비교

`schema_phaseB` 마이그레이션(`ALTER TABLE ... ADD COLUMN small_return`) → `grid.py` 216조합 재실행(2분22초) → `walkforward.py` Layer 2 재판정, 순서로 서버에서 실행.

### 3-1. `ex22_alpha` 변화폭 (overlay_freq별, |Δ| 평균)

| overlay_freq | 평균 \|Δex22_alpha\| | 최대 \|Δ\| |
|---|---|---|
| monthly | 0.089 | 0.148 |
| quarterly | 0.098 | 0.388 |
| semiannual | **0.911** | **1.613** |

`net_cagr`(전략 자체의 절대 성과, base 비교와 무관)는 Δ=0 — 이번 수정은 tilt 자체의 실행 로직이 아니라 **always-on 비교군만** 바꿨다는 것을 재확인.

monthly도 완전히 0은 아니다(예상보다 컸음) — `execution_date`가 `next_trading_day()`로 1거래일 지연되는 v0.3 "정밀" 설계 때문에, 월간 빈도에서도 `base_return`이 Phase A의 달력월 수익과 미세하게 다른 타이밍을 갖는다. 다만 quarterly/semiannual의 구간 불일치(수개월 vs 1개월)에 비하면 훨씬 작다.

`SPEC_08_B05` §5-3이 "특히 semiannual이 사실상 상수 배분에 가까운지 확인"을 "이 실험의 백미"로 꼽았는데, 그 답이 여기서 이미 나왔다 — **semiannual의 완벽한 100% 통과율은 신호가 아니라 구간 스냅샷 버그였다.** 버그가 없었다면 애초에 실험할 "타이밍 알파"조차 존재하지 않았다.

전부 `ex22_alpha`가 0.0006~0.0071 — 사실상 잡음 수준이고, `period22_share`가 92~99%라 애초에 C2(단일 에피소드 의존)에서 걸러진다. **살아남는 조합이 없다.**

## 4. 결론

**2026-07-10 리포트의 핵심 결론("68/144 통과, D 26/72·F 42/72")은 철회한다.** 원인은 신호나 전략이 아니라 always-on 비교군 계산의 구간 불일치 버그였다. 수정 후에는:

- Layer 2를 통과하는 조합이 **없다** (0/144)
- `ex22_alpha>0`인 소수 조합도 92~99%가 #22 한 에피소드에 의존 — C2에서 이미 탈락
- K를 키울수록 결과가 나빠짐 — 신호 반응성 자체가 역효과

## 5. SPEC_08_B05(타이밍 vs 탈집중 분해)에 대한 함의

`SPEC_08_B05`는 "현재 알파 = 타이밍 + 탈집중"이라는 전제로 설계됐는데, 이번 수정 후엔 애초에 분해할 만한 알파(ex22_alpha)가 거의 없다. `SPEC_08_B05` §7 자체의 사전 합의된 분기표에 대입하면:

- (가) 타이밍 실재: `timing_alpha_ex22>0`이면서 부호 안정 — 근거 조합 자체가 없어 성립 불가
- (나) 탈집중: `timing_alpha_ex22≤0` — 사실상 이쪽이지만, "탈집중 효과"라고 부를 만큼의 양(+)의 `deconc_alpha`조차 없을 가능성이 높음(always_on 대비 전체 alpha 자체가 대부분 소멸)

즉 `SPEC_08_B05`를 원래 스코프대로(6개 축 그리드 전체에 대해 정교하게 분해) 실행할 실익이 낮다 — **분해할 신호가 남아있지 않다.** 이 결과를 사용자에게 보고하고, B-0.5를 그대로 진행할지 / 축소해서 확인만 할지 / SPEC_08 Phase B 전체를 여기서 접을지 결정이 필요하다.
